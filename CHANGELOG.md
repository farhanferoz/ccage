# Changelog

All notable changes to ccage. Format follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/). Versions follow [Semantic Versioning](https://semver.org/).

## [0.14.0] — 2026-07-20

### Fixed — `ccage handoff` needed a manual `CLAUDE_CONFIG_DIR=` prefix to find any session
- **`ccage handoff` run from a plain shell looked in `~/.claude`, the one config dir that holds no cage sessions.** The `claude()` wrapper exports `CLAUDE_CONFIG_DIR` with `local -x` precisely so it never leaks into the shell, and `bin/ccage` never sourced the wrapper — so the CLI had no idea how cages are keyed and defaulted to the master dir. Every invocation needed a manual `CLAUDE_CONFIG_DIR=…` prefix, which is only possible if you already know the answer. `bin/ccage` now sources `claude-isolation.sh` (installed alongside the CLI's other libs) plus a user's `claude-overrides.sh` companion — the latter matters because that file is where `_CCAGE_OVERRIDE_ACTIVE=1` is set, so skipping it would silently ignore a user's `_ccage_config_dir_override`.
- **Resolution order:** `--config-dir DIR` (new flag) → `$CLAUDE_CONFIG_DIR` (unchanged when set, so no regression) → `_ccage_config_dir_for`, which brings `CCAGE_ROOT`/`CCAGE_PREFIX`/`CCAGE_SLOT`/override support with it → a scan of every cage for an `.owning_path` naming this project. The last two require the cage to actually hold sessions for the project's slug, so a cage that exists but has never run there falls through to the scan that finds the sibling slot which has. Several cages can own one path — that is what `CCAGE_SLOT` is for, and three project paths do so on the author's machine — so the scan picks the most recent session and prints which cages matched and which it chose. Failure names the cages actually searched and points at `--config-dir`, instead of blaming `~/.claude`.

### Fixed — pricing tables had drifted 3× on Opus, and every model newer than 2026-05-16 fell through to it
- **Four hand-maintained parallel rate tables drifted independently.** By 2026-07-20 the Opus row read $15/$75 against an actual $5/$25, every model released after the table's `# updated:` stamp fell through to the Opus default (costing a Sonnet 5 session at 5× reality), and all cache-write was charged at the 5-minute 1.25× rate even though these sessions cache at the 1-hour 2× TTL. Combined, one session measured at ~$41 actual was reported as ~$114. Each file now keeps **one** input/output table and derives every cache rate from it, so the rates cannot drift within a file; model patterns are globs, so a suffixed id such as `claude-opus-4-8[1m]` matches its family instead of the default. The per-TTL split is read from `usage.cache_creation.{ephemeral_5m,ephemeral_1h}_input_tokens` and priced separately, with a fallback to the cheaper 5-minute rate for transcripts predating the split.
- **This also lowers the `claude -r` resume cost estimate by 2–4×**, so with `CCAGE_RESUME_PROMPT_MIN_USD` at its default the resume prompt fires less often. Intended, not a regression.

### Fixed — teammate and background-task notifications were counted as user prompts
- **A fan-out session's brief was mostly notifications.** A teammate's idle message and a background-task completion are injected as `type: "user"` records with plain-string content, no `isMeta` and no `toolUseResult` — structurally identical to something the human typed, so every filter passed them. On one real session **7 of 8** extracted "prompts" were notifications, inflating `Turns: N user` to match and taking 5650 of the brief's 9430 bytes. They are now split into their own bucket rather than dropped, and the test that matters is the opposite direction: the predicate is **anchored at the start of the record**, so a human who merely *quotes* `<teammate-message ` — pasting a transcript excerpt, or asking about this very code — keeps their prompt. An unanchored match deleted it outright, which is the wrong failure direction for a tool whose job is recovering the human's own words: over-counting is recoverable by reading, deletion is not. The predicate is a single jq `def` used at all three extraction sites, so the production collector and the unit-test helpers cannot drift apart.

### Added — `ccage handoff` reports delegated (subagent) work
- **Token totals and the cost estimate now include every subagent.** The brief read only the main transcript, so a fan-out session reported its orchestrator's spend and nothing else — measured on one session, 800 KB of main transcript against 7.4 MB across four subagents, and $8.19 reported against $44.66 actual. The header gains an "of which delegated" split and a delegated turn count. Each agent is priced at the model that actually served its turns rather than a session-wide rate, since agents run on different tiers.
- **New `## Delegated work` section** — one row per agent (name, type, model, turns, cost, files, how its last turn ended), read from `<session-id>/subagents/agent-*.jsonl` and the `.meta.json` sidecar. The terminal state comes from the agent's own last assistant record, so the weekly-limit case shows up here as `api error: …` instead of leaking into the user-prompt list. The wording is deliberately weak — `ok` means the last turn completed normally, not that the agent finished its task, because a transcript carries no explicit termination record. `## Files touched` gains a `By` column merging the main thread and every agent; a path touched by several sources collapses to a count so the column cannot run away. `--no-subagents` restores the previous main-transcript-only view.
- **Size is the binding constraint**, because the brief is the input to a paid turn. Every section is now bounded: the agent list is capped at the 12 costliest, the commands listing is flattened to one capped line per command, and — the section that actually blew the budget — **user prompts are capped by length and by a section byte budget, not only by count**. Measured across every real session on the author's machine, 2 of 25 briefs exceeded the 15 KB target and one reached **77 KB**, almost entirely pasted prompt text; that case now composes to 6.7 KB. The commands listing was separately *wrong*: a heredoc or multi-line python block arrived with its newlines intact, so each of its lines became its own bullet.
- **Pasted prompt text can no longer forge the brief's structure.** Prompts were interpolated verbatim, so a pasted prompt carrying its own `## ` headings produced 30 bogus sections interleaved with the real ones — and the model reading the brief cannot tell prompt text from section structure. Line-leading `#` is now escaped, and every cell in the delegated-work table is pipe-escaped so an API-error message containing `|` cannot split the row.
- **An unknown model is now labelled as a guess.** It is still priced at the current Opus tier, but the brief says so outright instead of printing "standard rates", which read as an assertion that ccage knew the price. A delegating session's total likewise no longer names a single model beside a figure that spans tiers.
- Measured: a real four-agent session composes to 8.7 KB in 0.4 s; a real recovery brief to 12.1 KB.

### Added — `/checkpoint --from-session`: recover a session that ended without a checkpoint
- **The case the continuity loop could not cover.** A session that crashed, hit a limit, or was simply closed left no `RESUME` update — but its transcript is retained in full on disk. `--from-session` runs `ccage handoff --stdout` against the previous session (the newest transcript is the live one, so it takes the next-newest unless given an id), reads the brief, and merges it into `RESUME.md` in place. The split of responsibility is what keeps recovery cheap: `ccage handoff` reads and compresses with zero API calls, the skill applies judgment, and only ~10 KB is ever read by the model.
- **The skill must report what did *not* recover.** A transcript records actions, not reasoning: files touched, commands run, prompts given, which agents ran and how they ended all reconstruct well, but decisions and rejected alternatives are gone unless a prompt happened to state them. Anything written into `### Decisions` from a recovered brief has to be marked inferred. Manual and opt-in by design — nothing runs at session start or end.

### Fixed — `ccage handoff` did not see workflow subagents at all
- **A Workflow-spawned agent lands one directory deeper than a plain subagent, and the glob was not recursive.** Plain subagents live at `<session>/subagents/agent-*.jsonl`; a workflow's live at `<session>/subagents/workflows/wf_<id>/agent-*.jsonl`. Every one of the latter was missing from the token totals, the cost estimate and the delegated table — the same under-reporting the delegated-work section was added to eliminate, still live in the layout the Workflow mechanism uses. **580 such transcripts existed on the author's machine.** The silent case is the damaging one: a session that delegated *only* via a workflow has no top-level agents, so the brief rendered no delegated section and read as though nothing had been delegated. Measured across the six affected real sessions, cost corrections ran from +$5.78 to **+$77.16** (one session reporting $73.45 against an actual $150.61); the two zero-top-level sessions reported $70.90 and $97.59 with no delegated section at all.
- **Workflow agents collapse to one row per workflow, not one per agent.** A workflow fans out by construction — 166 agents in one real session — and a row each would have added ~15 KB to a single brief, against a 15 KB whole-brief target. Grouping keeps the growth to 146–530 bytes per brief while absorbing up to 166 agents, with the file count reported as the distinct path set across the workflow. The layout question was then closed exhaustively rather than by example: enumerating every path shape under every cage's `subagents/` confirms exactly two transcript layouts across all 3,184 agent transcripts, and both are now read.

### Fixed — pasted and transcript text could still forge the brief's own sections
- **Prompts were neutralized; the section immediately after them was not.** `## Last assistant turn` was word-capped but interpolated verbatim, and assistant turns routinely sign off with their own `## ` headings — **13 of 120 real sessions** produced a brief whose last-turn section contained one, structurally indistinguishable from `## Files touched`. The escape is now a single definition applied at both sites rather than two copies that can drift, taking that figure to 0 of 120.
- **The `## Files touched` table is assembled in `awk`**, so the pipe-escape guarding the delegated-work table was unreachable from it. A path containing `|` — legal on Linux and macOS — produced a row with 7 cells against a 5-cell header. Both tables now share one escape definition.
- **The prompts heading counted what was offered, not what rendered.** It was emitted before the section byte budget ran, so 50 prompts at the default `--max-prompts 20` rendered 14 under a heading promising 20. It is now emitted after the pass.

### Fixed — `ccage-auto --set` was a silent no-op, and a new session could wipe a live one's thresholds
- **Three parsing failures.** Control subcommands were recognised only at `argv[0]`, so `ccage-auto-yolo` — which expands to `ccage-auto --dangerously-skip-permissions --set soft=40` — never matched, and the "first unrecognised token means the rest belongs to `claude`" rule forwarded `--set soft=40` to `claude`. They are now recognised anywhere in argv, `--set soft=N hard=M` is accepted at launch as equivalent to `--soft N --hard M`, and anything `--set`/`--soft`/`--hard` shaped warns rather than being silently forwarded.
- **A fourth failure had nothing to do with parsing, and was the one users actually hit.** `resume_autoload.sh` scoped its stale-state clear to the **project directory**, not to a session — so any new session starting in that directory deleted the control file a *different, still-running* session's watcher depended on. Reproduced from the logs: `--set soft=45` took effect, an unrelated session was born in the same directory 104 seconds later, and `.ccage-autock.conf` was gone 4 seconds after that, with the next nudge firing against the launch 35%. `ccage-auto` now records its pid and start time and the hook clears only when no other live watcher owns the directory.
- **Each watcher records itself in its own file, because two watchers in one directory reopened the same bug.** The first cut wrote a single `.ccage-autock.pid`: last-writer-wins on start, and removed unconditionally on exit. Two sequences were reproduced, both ending with a live watcher's state deleted and the user-visible symptom identical to the original complaint (a `--set` threshold silently reverting) — a second watcher starting in the directory and then exiting *normally* deleted the first's ownership, and a second watcher killed with `SIGKILL` left a record naming a dead pid, which is all the hook could see. Concurrent watchers in one directory are not hypothetical; two were found in this repo's own directory. Records are now `.ccage-autock.pid.<pid>`, one per watcher, and the hook blocks the clear while **any** recorded pid is live: no watcher ever writes or removes a sibling's file, so both failures are structurally impossible rather than narrowed. Records are written to a temp file and renamed (a hook reading mid-write no longer sees a truncated record and concludes there is no watcher), dead records are swept on the next watcher's start, and the legacy unsuffixed name is still read so a watcher started before the upgrade keeps its protection.
- **That liveness check reads one recorded pid via `kill -0` rather than scanning processes.** The first implementation used `pgrep -f` plus a per-match `lsof` and cost 1420 ms against a 34 ms baseline, and matched any process merely *mentioning* `ccage-auto` in its argv — including a shell grepping for it. Worst case is now 23 ms, and the false-positive class is gone by construction rather than patched around. A recorded start time guards against pid reuse; a missing or unreadable pidfile falls back to the previous unconditional clear.

### Fixed — a session that checkpointed could be left parked, never cleared
- **The watcher discarded a checkpoint it had already been given.** In the nudged state the occupancy-below-soft cancellation was evaluated *before* the confirmation check, so anything that put occupancy under the live soft threshold — most easily a `--set soft=N` raise mid-cycle — cancelled the cycle and returned to NORMAL even though the model had already checkpointed. Observed on 0.13.1: nudge at 35.1%, the model wrote its checkpoint and printed the sentinel, `--set soft=45` landed against a 38.2% reading, and the cycle was cancelled 0 seconds later. Nothing further was ever coming — the nudge tells the model to print the sentinel and stop — so the session sat idle indefinitely with its checkpoint written and no `/clear`, which reads as a hung session. The confirmation is now checked first: it requires the sentinel in text logged *since* the nudge, so it cannot fire on a stale arm or a manual `/clear`, and the cancellation still owns every unconfirmed case.

### Changed — auto-checkpoint thresholds re-anchored
- **Soft nudge now defaults to 40%, the hard backstop derives as `min(soft + 20, 100)`, and the re-nudge sits at `hard − 5`.** No new nudge stage — the same three messages that already existed, with their trigger points moved. The re-nudge was purely time-based and is now occupancy-anchored with the timer kept as a fallback, so a session that stalls below the soft threshold is still reminded. An explicit `--hard` always wins; the derived value warns when it reaches 100, since a backstop at a full window is too late to be useful. `clamp_thresholds` is the single choke point shared by launch flags, `--set`, and the watcher's live re-read.

### Fixed — `ps -o etimes=` is a Linux-only extension and failed silently on macOS
- **Elapsed process time is now read with `ps -o etime=` and parsed from `[[DD-]hh:]mm:ss`.** `etimes` (plural, seconds) is a procps extension absent from BSD/Darwin `ps`, and the failure was silent in both directions: the shell path skipped its guard entirely, so the watcher-liveness check would simply never fire on a Mac, and the Python path recorded no start token, degrading the pid-reuse guard to nothing. Caught before release only because a colleague running this repo on a Mac made cross-platform an acceptance criterion.

### Changed — CI now tests what it claimed to test
- **`bin/ccage` was never shellchecked.** It is now, alongside the `share/` and root scripts it was omitted from — this release changed it twice.
- **The macOS leg ran whatever `bash` PATH offered.** `shell: bash` resolves via PATH and may find a newer Homebrew bash, so a green macOS leg was not evidence of the stock bash 3.2 compatibility ccage targets. The suite now runs with `/bin/bash` first on PATH there — bats is `#!/usr/bin/env bash` and bats-core supports 3.2 and above, so harness and code under test both run on the platform's own bash. The version is printed rather than assumed.

## [0.13.1] — 2026-07-16

### Fixed — RESUME's plan-doc directive now keys on the `### Plan` section only
- **The resume autoloader's read-and-dispatch directive (shipped in 0.13.0) matched plan docs by filename across the *whole* RESUME, so it named the wrong docs — or fired on sessions that were never plan-governed.** `resume_autoload.sh` scanned every line for `PLAN*.md` / `IMPLEMENTATION_PLAN*.md` / `plans/…` / `MASTER.md` / `*-plan.md` shapes, which swept in tangential, stale, and cross-repo `PLAN.md` mentions still sitting in Session-block history. Observed live: a resume listed **four** plan docs — two of them from *other* repositories — and **missed** the single doc the session's own `### Plan` section named (`WEEKLY-LIMIT-GUARD.md`), because that filename didn't match the plan-name pattern. On a session with no plan in play at all, one historical `PLAN.md` mention was enough to fire the dispatcher directive spuriously. Detection is now scoped to the `### Plan` block — the one place `/checkpoint` records the governing doc's exact path — and is filename-agnostic there: any `.md` under `### Plan` that exists on disk is named; a RESUME with no `### Plan` section, or whose refs are all missing, emits nothing. The two halves of the feature now connect (the checkpoint skill writes the path into `### Plan`; the autoloader reads exactly that section instead of guessing by filename). Regression tests rewritten to the scoped contract, plus three over-fire guards: a plan-doc mention outside `### Plan`, a stray existing `PLAN.md` elsewhere, and the bootstrap placeholder all stay silent.

## [0.13.0] — 2026-07-16

### Added — weekly-limit floor for `ccage-auto` (`CCAGE_AUTOCK_WEEKLY_FLOOR`)
- **`ccage-auto` can now stand an unattended run down before it burns the account's seven-day usage limit, not just its context window.** The existing context watcher only ever managed occupancy; it was blind to the weekly cap, so a long autonomous run could exhaust the account's week with nobody watching. Armed with `--weekly-floor N` / `CCAGE_AUTOCK_WEEKLY_FLOOR=N` (`N` = weekly capacity **remaining**, in percent), a new stage in the same checkpoint-watcher poll loop warns at `floor + 5`% remaining, then at the floor itself forces a checkpoint via the existing hard-backstop machinery, appends a stand-down note to `RESUME.md`, and stops driving the session — no `/clear`, no resume; the run is left safe to close until the weekly window resets. Opt-in, off by default, same posture as `CCAGE_SESSION_DOCS` / `CCAGE_SEED_LOCAL_HOOKS`.
- **The sensor is the rate-limit state Claude Code already passes to every statusline render** — the only place local code can read the server-side seven-day percentage without credential handling (`ccusage` was considered and rejected: it estimates local token spend from transcripts, not the server-side figure). `share/hooks/ccage-statusline-tee.sh` wraps a cage's statusline command, tees `{five_hour, seven_day, ts}` atomically into `$CLAUDE_CONFIG_DIR/rate-limits-state.json` on every render, then execs the real statusline — fail-open at every step, so a missing `jq` or a malformed payload never breaks the status bar. `_ccage_seed_statusline_tee` (`share/claude-isolation.sh`) wraps/unwraps a cage's `statusLine.command` on every launch, keyed off the same env var, following ccage's UI-only seeding discipline.
- **A stale sensor (older than 30 minutes) or a missing one is treated as unknown, never as "safe" or "at the floor"** — no stand-down fires off a dead sensor, and none already armed is silently cancelled by one either. Remaining capacity climbing back above `floor + 5`% — a weekly-window reset, or a lowered floor — cancels any in-progress stage, including an unconfirmed floored stand-down.
- **`ccage-auto --status` and `ccage doctor` both report the feature's posture.** The former prints a `weekly floor :` line with a live sensor readout; the latter reports `disabled` or `armed at <N>% remaining — sensor state in <X> of <Y> scanned cage(s)` across every scanned cage. `ccage doctor --unseed` and `uninstall.sh` unwrap every cage's wrapped statusline before the tee script is deleted, so no cage is left pointing at a missing hook. `install.sh` deploys the tee to the fixed hooks path alongside the existing ask-guard hook.

Full design: `docs/WEEKLY-LIMIT-GUARD.md`.

## [0.12.0] — 2026-07-16

### Added — seed the user's own local hooks into every cage (opt-in: `CCAGE_SEED_LOCAL_HOOKS`)
- **`_ccage_seed_local_hooks` merges the hook registrations from the user's real `~/.claude/settings.json` into each cage's `settings.json`, so a policy they maintain globally actually applies inside every project.** A hook script under `~/.claude/hooks` is inert until some `settings.json` registers it, and every cage has its own — so ccage seeded only its session-docs hooks and a user's policy hooks reached a cage only if someone hand-edited it. Measured on the author's machine (2026-07-15): their per-turn orchestration gate was live in **34 of 71 cages**, 8 cages used within the last month had it nowhere, and **every new cage was born without it** — while the policy file itself claimed it fired in "every single project". The claim was false and nothing could have noticed. Runs every launch, so it self-heals and backfills existing cages. Ownership line held: ccage owns cage **wiring**, the policy **content** is the user's — this copies registrations and hardcodes no hook names, and skips ccage's own session-docs hooks (which have their own opt-outs). Docs: `docs/SEED-LOCAL-HOOKS.md`.
- **Seeding is judged per HOOK, not per matcher entry.** A matcher group carrying N hooks is split into N single-hook entries on the way in. The first implementation identified each entry by its *first* hook (`script_base(cmds[0])`), which made every hook after the first invisible: never named, never checked, never seeded — and, had the group ever been copied, it would have duplicated hook #1. Caught 2026-07-15 by a live dry-run: a new `commit_provenance_guard.sh` registered as a *second* hook on the existing `PreToolUse: Bash` group (beside `xpu-guard.sh`) reached **zero of 71 cages** while the companion backfill script reported **"already complete: 37"** — truthful about `xpu-guard`, silent about the new guard. Same species as this seeder's earlier tilde-form bug, and the same species as the defect the seeder exists to fix, one level up: **a control that is real in the file and absent in effect, reporting success.** Per-hook judgement also lets a group that mixes a ccage-owned hook with a user hook seed the user's half instead of being skipped wholesale. Three regression tests, each proven to fail before the fix.
- **`ccage doctor` gained a report-only line for this feature.** States whether `CCAGE_SEED_LOCAL_HOOKS` is enabled and, when it is, how many of the user's policy-hook registrations are present vs. missing across scanned cages — a read-only twin (`_ccage_doctor_local_hooks_status`) of the seeder's hook-derivation logic that never writes anything, cross-referenced with "KEEP IN SYNC" comments in both directions.

### Fixed — `ccage handoff`'s clipboard auto-copy could hang forever on an unanswering Wayland socket
- **File-mode `ccage handoff` (and `/handoff`) could hang indefinitely instead of writing the brief and returning.** `_ccage_handoff_copy_to_clipboard` tries `pbcopy` → `wl-copy` → `xclip` → `xsel`, piping the brief into whichever is found first. When `WAYLAND_DISPLAY` is set but the socket never answers (sandboxed shells, an SSH session carrying a stale env var), `wl-copy` blocks in the FOREGROUND before it daemonizes — no fd trick helps, since the hang happens before the process backgrounds itself. Observed live: one run wedged 23 minutes before being killed. Each copier invocation is now wrapped in `timeout 5` when GNU `timeout` is on PATH; absent (stock macOS) the bare call is unchanged, since `pbcopy` doesn't share this failure mode. Verified against a real unix-socket listener that accepts the connection and never responds: the old bare call hangs past an 8-second bound, the fixed call returns in ~5s.
- **Tests now stub `pbcopy`** so no test run can ever reach a real clipboard tool — it's first in the fallback chain, so a fast, always-succeeding stub on PATH ahead of everything else wins regardless of what's actually installed on the machine running the suite. The pre-existing fd-3-close trick is kept as belt-and-braces (it only ever covered the *daemonizing* wl-copy hang, not the foreground one that actually bit). New regression test asserts file-mode `generate` completes in well under 10 seconds with the stub present.

### Added — RESUME's plan-doc pointers are re-asserted with a read-and-dispatch directive
- **`resume_autoload.sh` now names every plan doc the injected RESUME references and exists on disk** (`plans/`/`plan/` contents incl. `MASTER.md` strands, `PLAN*.md`, `IMPLEMENTATION_PLAN*.md`, `*-plan.md`; capped at 5), followed by a directive: READ each doc before executing any task it governs, and treat an execution-level plan as DISPATCHER work (dependency waves), never a list to walk sequentially. Measured failure this closes (user-reported, recurring): a resumed session acts on RESUME's summary bullets, never opens the plan they point to — tasks silently drop — then executes sequentially, because a bullet list carries no dependency structure. RESUME is deliberately lean (budget-nagged), which makes its pointers load-bearing; a pointer to a missing file earns silence, not a directive. Five regression tests.

### Changed — softened the autonomous `AskUserQuestion` guard's deny message
- **The guard no longer asserts "the user is away."** `share/hooks/autonomous_ask_guard.sh` arms off `CCAGE_AUTONOMOUS=1` for the life of the `ccage-auto` process, with no way to detect if the user has since returned — so the old wording could be shown while they were actively typing. The message now says the run is autonomous and the tool must not block it, but tells the model to ask in prose instead of blocking if the user appears to be present (recent genuine user turns), keeping the reversible-default / `RESUME.md`-logging / batch-questions guidance for the truly unattended case. No logic changes — no presence detection was added; the guard's whole value is staying trivially correct. Documented in `docs/FEATURES.md`, including the arms-for-the-whole-session limitation this addresses.

### Added — `install.sh` backs up an existing target before overwriting it
- **`install_file()` now writes a `<target>.pre-update-<YYYYmmdd-HHMMSS>` backup before overwriting a file that already exists and differs from what's being installed.** `claude-isolation.sh` is sourced by every interactive shell, so a bad overwrite (a syntax error, an unwanted downgrade) previously broke every new shell with no way back. Skipped when the target doesn't exist yet (fresh install) or is byte-identical to the incoming file, so re-running install repeatedly doesn't litter backups.

## [0.11.3] — 2026-07-13

### Fixed — circuit breaker could falsely "verify" a stop/vouch it never got
- **A stop or vouch could be marked verified even when the model never sent one.** `scan_parent_transcript` matched the `CCB-VOUCH`/`CCB-STOPPED` compliance regexes against every transcript line regardless of who authored it. But the CB's own nudge/stop directive, injected over the pty, lands in the same transcript as an ordinary `"type": "user"` turn — indistinguishable from real input at that layer — and necessarily spells out the exact marker grammar as the example the model is meant to copy. So the very next poll after an injection, the CB's own instruction text alone satisfied its own verification regex, regardless of whether the model ever replied. Caught live: a real orchestrator explicitly refused to fabricate a `CCB-STOPPED` reply for a teammate it had confirmed (via `TaskList`) had already finished — yet the ledger still recorded `stop_verified` twelve seconds later, traced to the CB's own injected line, not the model's refusal text. Fixed by scoping `CCB-VOUCH`/`CCB-STOPPED` matching to genuine `"type": "assistant"` lines only. Regression test proves a `"type": "user"` line carrying the exact grammar (the injected-instruction shape) is correctly ignored.

## [0.11.2] — 2026-07-13

### Fixed — circuit breaker never actually tracked a real teammate
- **The stuck-subagent circuit breaker discovered zero teammates in every real Claude Code session, on every tier, since v0.11.0 shipped.** `list_subagent_transcripts` globbed `session_dir/subagents/`, but a real session nests subagent transcripts one level deeper, under a directory named for the session's own transcript-file stem (`session_dir/<session_id>/subagents/`). Every unit test's fixture placed transcripts at the flat (wrong) path, so this went uncaught until an attended live-fire run against a real session surfaced it: `.ccb-state.json` stayed `{"agents": {}}` after nine real minutes with a genuinely idle, correctly-formed teammate transcript sitting one directory below where the code looked. Fixed by threading `session_id` into the lookup. Regression test builds the real nested layout plus a same-named decoy at the old flat path to prove it isn't picked up by mistake; the two `test_autock.bats` live-fire fixtures (which drive a real `ccage-auto` process) are corrected to the same real layout. Same root-cause shape as the v0.11.1 installed-lib bug: a test fixture modeled a plausible-but-wrong directory layout. Still ships deployed at `observe` — this does not change the default rollout tier, only fixes discovery being silently broken underneath it.

## [0.11.1] — 2026-07-13

### Fixed — circuit-breaker lib resolution in the installed layout
- **The circuit breaker now actually loads when installed.** `bin/ccage-auto`'s `_load_ccb` and `bin/ccb-report` located the lib by matching any directory named `lib/`, so in an installed tree `<prefix>/lib` (e.g. `~/.local/lib`, Python's user site-packages parent) shadowed the intended `<prefix>/share/ccage/lib`: `_load_ccb` silently fell back to a no-op (breaker disabled) and `ccb-report` raised `ModuleNotFoundError`. Both now key on the actual module file (`lib/subagent_watch.py`), so a bare `lib/` decoy is skipped. ### Fixed (2026-05-23 publish attempt — surfaced by first GH CI matrix run)

Fixed at 2 call sites — `share/claude-isolation.sh:508` (`command claude "${_ccage_extra_args[@]}" "$@"`) and `share/ccage-handoff.sh:741` (`_ccage_handoff_generate "$jsonl" "${pass_args[@]}"`) — with the portable `${arr[@]+"${arr[@]}"}` guard. - **`touch -d` in tests is GNU-only.** `tests/test_handoff.bats:65-66` used `touch -d '2020-01-01'` to backdate fixture files. GNU touch accepts date-only strings; BSD touch (macOS) only accepts full ISO timestamps with `-d` (`YYYY-MM-DDThh:mm:SS[.frac][tz]`). Switched both calls to `touch -t [[CC]YY]MMDDhhmm` which is POSIX and accepted identically by GNU and BSD touch. Caught by `test_handoff.bats` test 22 failing on `macos-latest × bash` CI cell. Commit `d190cff`.

### Added
- **`claude -r` / `claude -c` cost interception (Phase 6b).** When the wrapper detects a resume invocation (`-c`, `--continue`, `-r <uuid-prefix>`, `--resume <uuid-prefix>`), it sums per-turn `cache_creation_input_tokens` from the session JSONL, subtracts an estimated 19K tools+system prefix that survives resume, computes a ±25% dollar range using the model's cache-write rate, and prompts `[r]esume / [h]andoff / [c]ancel`. Bare `-r` / `--resume` (no id) passes through because Claude Code may launch its own session picker we can't predict. Gates: `CCAGE_DISABLE=1`, `CCAGE_NO_RESUME_PROMPT=1`, non-tty stdin, or estimated cost below `CCAGE_RESUME_PROMPT_MIN_USD` (default `0.25`). `h` branch delegates to `ccage handoff` (installed by `install.sh`). Spec at `docs/PHASE-6-HANDOFF-SPEC.md`. 28 bats tests in `tests/test_resume_interception.bats` plus 3 e2e gate-assertions in `tests/validate-e2e.sh`.
- **`ccage handoff` — offline session brief generator.** Turns any session JSONL into a Markdown brief with user prompts, files touched, bash commands, last assistant turn, token totals, and a cost estimate. Zero API calls (pure shell + jq). Motivation: Claude Code's `claude -r`/`-c` always cache-miss the message prefix on resume (structural diff at `messages[0]`, not TTL — see GitHub #42309, #43657), so a fresh-session + handoff-brief workflow avoids the $0.50–$2 cache-rewrite tax per resume on long Opus sessions.
